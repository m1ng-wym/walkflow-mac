## AGENTS.md 维护规则

`AGENTS.md` 不是进度日志，只记录长期有效的项目规则和项目事实。

在长任务最终交付前，如果本次任务改变了以下长期事实或长期规则，必须检查并按需更新本文件：

- 技术栈、包管理器、项目形态或部署平台发生变化
- 核心目录结构、内容存放位置、数据模型或内容模型发生变化
- install、dev、build、lint、typecheck、test 或完整验证命令发生变化
- 安全和权限边界发生变化，例如允许引入后端、登录、统计、数据库、真实 provider 集成或部署流程
- 长任务文档规则、最终回复要求或文档语言规则发生变化
- 项目进入新的长期阶段，导致本文件中的目标、边界或默认假设不再准确

如果本次任务没有改变上述长期事实或长期规则，不要为了“保持更新”而修改本文件。

以下内容不要写入本文件；如果属于长任务，应写入对应 `docs/tasks/<task-name>/`：

- 单次任务进度
- 某次验证结果
- 临时设计方案
- 临时端口、部署 ID、一次性错误或短期阻塞
- 某个功能或文档的临时写作状态

如果发现本文件与当前代码、命令或项目事实不一致，先核验事实。若差异属于本次任务范围且会长期有效，应同步更新本文件；否则在当前任务的 `review.md` 记录差异、暂不更新原因和后续处理建议。

## 安全和权限

以下操作必须先问用户：

- 删除文件
- 移动文件
- 执行不可逆或难以恢复的改动
- 大幅修改依赖
- 修改公共 API、数据模型、内容模型、路由形态或核心架构边界
- 添加后端、登录、数据库、统计、管理后台或 provider 集成
- 把项目改成 monorepo 或新增子项目
- 修改生产配置
- 触碰域名、DNS、账单或部署设置
- 调用真实外部服务或真实 provider
- push
- deploy
- 破坏性 git 操作，例如 `git reset --hard`、`git clean`、强制覆盖用户改动、强制推送、回滚或删除未确认的工作

默认不要永久删除文件。如果用户批准移除，尽量移动到废纸篓。除非用户明确要求，不要使用 `rm -rf`。

本项目允许在任务范围内直接创建本地 commit，不需要每次额外询问用户。commit 前仍必须确认改动范围只包含当前任务相关文件，并避免把密钥、隐私数据、一次性本地状态或用户未要求纳入的文件加入提交。`push`、`deploy` 和任何破坏性 git 操作仍必须停止并等待用户明确确认；破坏性 git 操作默认不做。

无需额外询问即可执行：

- 只读检查
- 请求范围内的非破坏性本地编辑
- 请求范围内的本地 commit
- 仓库已有命令所定义的本地 lint、typecheck、test、build 或等价验证
- 为验证启动本地 dev server

如果启动了 dev server，应报告 URL，并避免不必要地留下长期运行进程。

不要把密钥、账号标识、一次性 token、私有凭据或临时本地状态写入本文件。

## 任务状态和文档

只为长任务创建任务文档。

长任务包括：

- 项目初始化或架构工作
- 核心目录结构、数据模型、内容模型或公共 API 变更
- 共享组件、共享工具、构建配置或验证流水线变更
- 多文件功能开发
- 大幅依赖变更
- 会影响后续实现决策的调研

普通文本编辑、小型文档调整和简单 Git 或仓库维护操作默认不要创建任务文档。例如配置 remote、改分支名、首次 push、更新 `.gitignore`、新增占位 README。只有当这类操作改变长期项目规则、工程流程、部署流程或验证标准时，才需要更新 `AGENTS.md` 或既有任务文档。

长任务状态写入：

```text
docs/tasks/<task-name>/
```

每个长任务固定使用四个文件：

```text
task.md
plan.md
progress.md
review.md
```

文件含义：

- `task.md`：目标、背景、范围、非目标、用户约束
- `plan.md`：实现计划、阶段、顺序、当前优先级
- `progress.md`：已完成内容、当前状态、下一步、阻塞
- `review.md`：风险、review 发现、验证命令和结果、跳过的检查、关键决策

本项目不要创建 `test.md` 或 `decisions.md`。

开始或恢复长任务时：

1. 读取本文件。
2. 如果任务目录已存在，读取该目录下的四个文件。
3. 如果涉及特定组件、模块、内容体系或已有任务文档，读取对应现有文档。
4. 如果文档和代码不一致，先核验当前仓库事实，再编辑。

长任务执行过程中，应在大范围编辑前、每个有意义的阶段后、遇到阻塞时、暂停前和最终交付前更新任务文档。

长任务最终交付前，必须做自审，并把风险、验证结果、跳过的检查和关键决策记录到 `review.md`。
