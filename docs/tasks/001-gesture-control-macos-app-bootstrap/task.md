# WalkFlow-Mac 任务说明

## 目标

构建 `WalkFlow-Mac`：一个 macOS AppKit 原生工具，通过前置摄像头识别用户在 1 到 2 米距离内的手势，实现完整远距控制能力：屏幕滚动、快捷键触发、识别暂停/恢复、误触保护和状态反馈。

## 背景

用户在 vibe coding 时希望减少久坐，不想一直坐在屏幕前敲键盘或操作鼠标。目标使用姿态是站立或走动，并能通过对着 Mac 前置摄像头做手势来观察编码进度、滚动屏幕、触发语音输入或其他按键。

## 范围

- 正式项目名为 `WalkFlow-Mac`。
- GitHub 远端仓库为 `https://github.com/m1ng-wym/walkflow-mac.git`。
- Git 默认主分支为 `main`。
- macOS 原生 App。
- 使用摄像头进行手势识别。
- 支持 1 到 2 米远距离交互。
- 第一版覆盖完整控制面，而不是只做单一滚动或单一快捷键。
- 需要包含用户可理解的状态反馈和误触保护机制。
- 暂停/恢复识别采用混合控制：菜单栏和全局快捷键作为硬开关，远距手势作为软开关。
- 第一版手势交互采用准备姿态模型：用户摊开手掌面向摄像头并短暂停顿后进入可控制状态。
- 准备姿态触发后开启控制窗口，窗口内滚动手势和指定快捷键触发手势有效。
- 控制窗口退出条件包括：超过约 5 秒没有新动作指令、手部离开画面、用户做停止手势。
- 准备姿态手势为五指张开、手掌面向摄像头。
- 控制窗口内滚动手势采用静态指向姿态：一指向上且其余四指收缩代表屏幕向上滚动；一指向下且其余四指收缩代表屏幕向下滚动。
- 停止手势采用握拳：五指全部收拢时退出控制窗口或停止当前连续滚动。
- 滚动不采用大幅上/下挥手作为默认动作。
- 滚动触发采用短按/长按混合：一指向上/向下姿态稳定约 0.2 到 0.4 秒先滚动一段，继续保持超过阈值后进入连续滚动。
- 连续滚动期间如果手势变化，应立即停止滚动。
- 快捷键触发的第一版目标键为键盘右侧 `Command` 键，用户当前用它作为语音输入切换键：按一次开始语音输入，再按一次结束语音输入。
- 右侧 `Command` 键由 `OK` 捏合手势触发：拇指和食指形成圆形接触，其余手指张开。
- `OK` 捏合手势应被解释为一次性点击事件，不能因为持续保持姿态而连续触发右侧 `Command`。
- `OK` 捏合触发规则采用稳定触发加冷却：姿态稳定约 0.2 到 0.4 秒触发一次，然后进入约 1 秒冷却；冷却后如果仍保持 `OK` 不再触发，必须松开后重新做 `OK` 才能再次触发。
- App 外壳采用主窗口为主、菜单栏状态为辅：主窗口显示摄像头预览、手势状态、权限和设置；菜单栏提供快速开关和其他配置项入口。
- 后续开发全流程必须使用 AppKit，不可以使用 SwiftUI。
- 主窗口采用左右分栏，比例约 `1:4`：左侧为配置、权限、HUD 和快捷键；右侧为摄像头预览和实时识别。
- 右侧区域内部再分区：摄像头预览为主要部分，实时识别反馈复用浮动面板样式，并固定浮动在摄像头预览区域右上角。
- 权限体验采用主窗口权限面板常驻加启动时一次性引导：启动时检查并引导必需权限，主窗口左侧常驻显示 Camera、Accessibility 和可能的 Input Monitoring 权限状态与修复入口。
- 缺少必需权限时，App 主窗口仍可打开，但控制能力进入阻塞状态，HUD 显示红点和权限图标。
- 菜单栏图标显示总状态，并提供 `Enable`、`Pause`、`Show HUD`、`Open Window`、`Settings`、`Quit`。
- 硬开关第一版不预设全局快捷键，但在设置中保留可配置全局快捷键的选项。
- 手势识别技术方案必须满足免费使用和轻量运行两个硬约束，后续不能引入需要付费订阅、云端推理或高内存/高 CPU 常驻负载的方案。
- 手势识别主线架构采用 `AVFoundation + Apple Vision hand pose + 自研轻量 GestureClassifier + GestureStateMachine`：AVFoundation 采集前置摄像头帧，Vision 输出手部关键点，自研几何规则识别五指张开、一指向上、一指向下、握拳、OK 捏合，状态机处理准备姿态、5 秒控制窗口、短按/长按滚动和右侧 `Command` 冷却。
- 系统事件注入采用 `CGEvent` / Accessibility 路线，用于屏幕滚动和右侧 `Command` 键触发。
- MediaPipe 作为备选验证方向：仅当 Apple Vision 在 1 到 2 米距离下对 OK 捏合或一指方向等关键手势识别不稳定时，才进行 MediaPipe 原生 spike；只有 MediaPipe 显著提升稳定性且 CPU/内存可接受，才纳入运行时。
- Roboflow `supervision` 不进入第一版 App 运行时；它可作为离线研究或视觉算法原型参考。
- 状态反馈采用可固定浮动面板：面板可 pin 住并保持固定浮动在屏幕右上角，不因点击其他区域而消失；面板用简单图形展示上滚、下滚、触发右侧 `Command`、冷却、暂停等状态。
- 浮动面板可拖动并记住位置：默认出现在屏幕右上角，用户可拖动以避开特定 App UI，下一次启动恢复上一次拖动后的最后位置。
- 浮动面板同一时间只显示一个大图标，不同时显示多个小图标，避免远距观看时杂乱。
- 浮动面板外观参考用户草图：小巧、内容简洁，顶部居中有一个小箭头，形成从菜单栏拉下来的视觉感；面板中心区域显示当前状态图标，用户草图中的方框仅表示图标位置，实际 UI 不显示该方框。
- 浮动面板左上角显示一个状态点：`Disabled`、权限缺失、手部丢失、停止等阻塞/不可控状态显示红点；`Standby` 和正常可用状态显示绿点。
- `Standby` 状态不显示中心图标，中心区域保持空白，仅通过左上角绿点表达可用待机。
- 中心状态图标使用 `https://useanimations.com/#explore` 的动效图标，必须保留网站原样动效；后续实现应使用其 Lottie JSON 动画资源，而不是静态图或重新绘制图标。
- `Disabled` 使用 `Lock / Unlock` 图标：硬开关关闭时显示 lock 状态；开启硬开关后触发一次 unlock click 动效，动效结束后进入 `Standby`，中心图标消失。
- `Permission` 使用 `Alert triangle` loop 图标显示，动效保持。
- `Ready` 使用 `Infinity` loop 图标显示，动效保持。
- `Scroll Up` 使用 `Arrow up` loop 图标显示，动效保持。
- `Scroll Down` 使用 `Arrow down` loop 图标显示，动效保持。
- `Command` 使用 `Dribbble` hover 后的动效循环显示；第一次触发右侧 `Command` 后保持该动效，直到下一次触发右侧 `Command` 结束语音输入。
- 第二次触发右侧 `Command` 结束语音输入后，不回到 `Ready` 图标；如果手仍可识别则回到 `Standby` 空白绿点，如果手不在画面则显示红点阻塞。
- `Hand Lost` 和 `Stop` 不显示中心图标，中心区域空白，左上角状态点显示红点。
- `Paused` 和 `Cooldown` 暂不显示中心图标，避免状态图标过多导致杂乱。
- Lottie 动效集成选择 `AppKit + 原生 Lottie macOS renderer`，不使用 SwiftUI，不使用 WebView 作为默认实现路径。
- 原生 Lottie 实现必须尽量复刻 useAnimations 网站的原样动效和播放控制；后续验证需要做视觉对比，不能用静态图替代。
- 后续开发使用 Build macOS Apps 插件能力。
